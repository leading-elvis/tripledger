## Problem Statement

旅行時記帳是最煩人的事之一。你在日本吃完拉麵，拿到一張全日文收據，要手動輸入金額、換算幣別、選分類、選分帳方式 — 這個流程讓人不想記帳。TripLedger 已經有台灣電子發票 QR Code 掃描和基礎 OCR 功能（中文收據），但出國旅行時面對日文、韓文、泰文收據就無能為力了。

目標：拍一張任何常見旅遊國家的收據照片 → AI 自動解析金額、店家、分類、幣別 → 預填建帳表單 → 用戶確認付款人和分帳方式 → 一鍵建帳。

## What Makes This Cool

「拍了就建帳」— 在異國旅行時，面對看不懂的收據，拍一張照片就能自動識別所有資訊並建好帳。這是讓朋友旅行時真的會說「哇」的功能，也是展示 AI + 多語言 OCR 技術整合能力的最佳 showcase。

## Constraints

1. **成本控制**：前期資金有限，必須使用免費或低成本方案。Google Cloud Vision API 免費額度 1000 次/月。預估使用量：5-10 位活躍用戶 × 每日 2-3 次掃描 × 旅行天數 5-7 天 ≈ 50-210 次/月，在免費額度內。若超出免費額度，降級到 ML Kit only 模式並提示用戶。
2. **已有基礎設施**：必須建立在現有 OCR 模組（ML Kit + Vision API + text-parser + brand-lookup）之上，不重造輪子。
3. **語言範圍**：先專注台灣人常去的旅遊國家 — 日本（ja）、韓國（ko）、泰國（th）、越南（vi）、英語系國家（en）。
4. **流程設計**：AI 預填 + 用戶確認，不做完全零操作（付款人和分帳方式必須人工確認）。
5. **入口**：App 內拍照 + 相簿匯入都要支援。
6. **圖片大小**：前端拍照/選圖後，壓縮至最長邊 2048px、JPEG 品質 85% 後再上傳（現代手機照片 5-10MB，壓縮後約 200-500KB）。

## Approaches Considered

### Approach A: 多語言 ML Kit 升級（最小變動）

ML Kit 加入日文/韓文腳本辨識 + 多語言 text-parser + 幣別自動偵測 + 簡化前端流程。

- **Effort**: S
- **Risk**: Low
- **Pros**: 完全免費、離線可用、改動最小
- **Cons**: 日文/韓文收據準確率不穩定、無法處理複雜排版
- **Reuses**: 現有 ML Kit 整合、text-parser、add_bill_page prefill

### Approach B: Google Vision API 多語言強化（理想架構）

Vision API language hints 擴展 + 智慧路由（本地優先、雲端補強）+ 旅行場景幣別推斷 + 一鍵掃描流程。

- **Effort**: M
- **Risk**: Low
- **Pros**: 準確率高、已有 Vision API 整合只需擴展、免費額度充足、架構未來可升級到 LLM Vision
- **Cons**: 需要網路連線（但旅行時通常有網路）
- **Reuses**: vision-ocr.service.ts、text-parser.service.ts、brand-lookup.service.ts、ocr_provider.dart、scan_receipt_page.dart

### Approach C: LLM Vision 全智慧方案（創意方向）

GPT-4o-mini Vision 直接解析收據圖片，返回結構化 JSON。

- **Effort**: M
- **Risk**: Med（API 成本、延遲較高）
- **Pros**: 準確率最高、能理解上下文、直接輸出結構化數據
- **Cons**: 每次 $0.01-0.03、延遲 2-5 秒、依賴外部 API 穩定性
- **Reuses**: ai-suggestion.service.ts、S3 上傳

## Recommended Approach

**方案 B: Google Vision API 多語言強化**

理由：你已經有完整的 Vision API 整合（vision-ocr.service.ts），只需要擴展語言 hints 和解析器。成本控制在免費額度內（1000 次/月），準確率遠高於純本地 ML Kit。架構設計保留升級到 LLM Vision 的空間，未來營收穩定時可以無縫切換。

### 具體實作計畫

#### 後端改動

**1. vision-ocr.service.ts — 多語言支援**
- 擴展 language hints：`['zh-Hant', 'zh-TW', 'ja', 'ko', 'th', 'vi', 'en']`
- 新增語言偵測邏輯：根據文字 Unicode 範圍判斷收據語言
  - 日文：Hiragana (U+3040-309F) + Katakana (U+30A0-30FF)
  - 韓文：Hangul (U+AC00-D7AF)
  - 泰文：Thai (U+0E00-U+0E7F)
- 根據偵測到的語言調整後續解析策略

**2. text-parser.service.ts — 多語言解析策略**

現有 text-parser 專注台灣格式。擴展方式：新增 `LanguageParserStrategy` 介面，每個語言一個實作類別，由 text-parser 根據偵測到的語言委派：

```
TextParserService
  ├── TaiwanParserStrategy (現有邏輯抽出)
  ├── JapaneseParserStrategy
  ├── KoreanParserStrategy
  ├── ThaiParserStrategy
  └── EnglishParserStrategy
```

各語言金額模式：
- 日文：`¥`, `円`, `税込`, `合計`, `小計`
- 韓文：`₩`, `원`, `합계`, `소계`
- 泰文：`฿`, `บาท`, `รวม`
- 英文：`Total`, `Subtotal`, `Grand Total`

各語言日期格式：
- 日文：`令和X年X月X日`, `2024年3月15日`
- 韓文：`2024년 3월 15일`
- 英文：`MM/DD/YYYY`, `DD/MM/YYYY`

**3. 新增 currency-detector.service.ts — 幣別自動推斷**

信心分數範圍 0.0-1.0，對應 UI 顯示：綠色 ≥ 0.8、黃色 0.5-0.8、紅色 < 0.5。

推斷規則（按優先級，每層疊加信心）：
- 優先級 1：**明確貨幣符號**（₩→KRW 信心 0.95, ฿→THB 信心 0.95）
- 優先級 2：**模糊貨幣符號 + 語言消歧**
  - `¥` + 日文語言偵測 → JPY（信心 0.9）
  - `¥` + 中文語言偵測 → CNY（信心 0.9）
  - `¥` 無語言線索 → 依 Trip 幣別設定（信心 0.6）
  - `$` 永遠不單獨推斷 → 需結合語言或 Trip 幣別（信心 0.5-0.7）
- 優先級 3：**語言 → 預設貨幣**（ja→JPY, ko→KRW, th→THB，信心 0.7）
- 優先級 4：**Trip 設定幣別**（信心 0.5，作為最終 fallback）

**4. brand-lookup.service.ts — 多語言品牌查詢**
- 擴展品牌映射表支援日文/韓文店名
- 新增常見連鎖店品牌：ローソン→Lawson, セブン→7-Eleven, ファミマ→FamilyMart
- 韓文：편의점, CU, GS25 等

#### 前端改動

**5. scan_receipt_page.dart — 簡化掃描流程**
- 新增「快速掃描」模式：大按鈕，拍照後直接進入結果頁
- 支援相簿選擇（已有 image_picker）
- 掃描中顯示動畫 + 進度提示

**6. scan_result_page.dart — 智慧結果展示**

資訊層級（由上到下）：
```
┌─────────────────────────────┐
│ [收據圖片預覽]                │
│  日文 (語言標籤，文字非emoji)  │
├─────────────────────────────┤
│ ❶ 金額 + 幣別 (最大字體)     │
│    ¥1,980 JPY  🟢 已確認     │
├─────────────────────────────┤
│ ❷ 店家名稱                   │
│    ローソン → Lawson          │
│    [品牌建議 chips]           │
├─────────────────────────────┤
│ ❸ 分類 + 日期                │
│    FOOD  3/15                │
├─────────────────────────────┤
│ ❹ 逐項明細（如果有）          │
├─────────────────────────────┤
│ ❺ [下一步：選付款人] CTA      │
└─────────────────────────────┘
```

信心指標設計（協作語氣）：
- 🟢 + 「已確認」— confidence ≥ 0.8，欄位正常顯示
- 🟡 + 「請幫我確認」— confidence 0.5-0.8，欄位可編輯，淡黃色背景
- 🔴 + 「需要手動輸入」— confidence < 0.5，空欄位等待用戶填寫
- 使用 AppTheme 的 success/warning/error 語義色彩
- 色盲友好：顏色 + 文字雙重提示
- Touch target ≥ 44px

學習回饋：用戶修改欄位後，顯示淡入淡出提示「已記住，下次更準確」(1.5秒後消失)

降級 banner：淡黃色 banner「離線模式，結果可能不完整」+ 所有欄位設為可編輯

**7. ocr_provider.dart — OCR 路由策略**

現有架構：前端 ML Kit 做快速預覽，後端 Vision API 做正式解析。**不合併兩者結果**（格式不同，信心分數不可比）。

路由邏輯：
- 前端 ML Kit 立即執行 → 顯示「掃描中」的即時預覽文字
- 同時上傳圖片到後端 → Vision API 做正式 OCR
- Vision API 結果回來後**取代** ML Kit 預覽結果
- **降級路徑**：如果 Vision API 失敗（網路錯誤、超出免費額度），顯示 ML Kit 結果 + 提示「雲端辨識不可用，結果可能不完整，請手動檢查」

API 契約變更：
- `POST /ocr/scan-receipt-image` request 不變（上傳圖片）
- response 新增欄位：`detectedLanguage: string`、`currencyResult: { currency: string, confidence: number }`
- 後端 `parseReceipt()` 和 `parseReceiptWithStructure()` 方法簽名需擴展，新增 `detectedLanguage?: string` 參數，用於委派到對應的 `LanguageParserStrategy`

#### 資料庫改動

**8. Prisma schema 遷移（必要）**
- `Bill` 新增 `detectedLanguage String?` 欄位 — 用於衡量 Success Criteria（各語言準確率追蹤）
- `CompanyBrandMapping` 新增 `nameJa String?`、`nameKo String?` 欄位支援多語言品牌名

## Success Criteria

1. 日文收據金額提取準確率 ≥ 85%
2. 從拍照到預填表單 ≤ 5 秒
3. 幣別自動偵測正確率 ≥ 90%
4. 用戶建帳操作步驟從 ~8 步減少到 ~3 步（拍照 → 確認 → 建帳）
5. 月 API 成本控制在 $0（免費額度內，預估 50-210 次/月 < 1000 次上限）
6. Vision API 失敗時，ML Kit 降級路徑正常運作，用戶可手動完成建帳

## What I noticed about how you think

- 你說「兩者都是」— 既要實用也要展示技術。這表示你不滿足於做一個 demo，你要做一個真正能用的東西。這種對品質的堅持會讓最終成品遠超「side project」的水準。
- 你選擇「先專注常見的」而不是「任何收據都能處理」— 你知道先做好一個範圍比什麼都沾一點來得重要。
- 你對成本的敏感度很明確 —「目前可用資金不高」直接影響了技術選型，這是務實的工程判斷。
- 你對「全自動但需要確認」的流程一點都不猶豫 — 你理解 AI 的極限，不追求虛假的完美自動化。
